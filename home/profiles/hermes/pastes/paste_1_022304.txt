LangGraph Workflow Orchestration Tool — run_workflow
Objetivo
Agregar una nueva herramienta MCP al servidor Olympus que permita a Hermes ejecutar flujos de trabajo dirigidos (grafos) entre Daimons, en lugar de orquestarlos manualmente turno por turno con talk_to. La herramienta se llama run_workflow.

Contexto y Motivación
Actualmente, cuando Hermes necesita coordinar múltiples Daimons (ej: "escribe código y luego audítalo"), debe hacerlo de forma imperativa:

talk_to(hefesto, open) → talk_to(hefesto, message) → talk_to(hefesto, poll)
→ talk_to(athena, open) → talk_to(athena, message, <resultado de hefesto>)
→ talk_to(athena, poll) → ...
Esto funciona para tareas simples pero tiene problemas críticos:

Fragilidad: Si Hermes pierde contexto (compresión, error), el flujo se rompe
Sin ciclos automáticos: Si Athena rechaza el código, Hermes debe "recordar" reenviarlo a Hefesto
Overhead de tokens: Cada paso consume tokens de contexto del propio Hermes
Con run_workflow, Hermes simplemente dice:

run_workflow(workflow="dev_and_audit", prompt="Implementa login con JWT en Python")
Y el grafo se ejecuta de forma autónoma, con ciclos automáticos (Hefesto ↔ Athena) y checkpoints.

User Review Required
IMPORTANT

Dependencia nueva: Este plan requiere instalar langgraph y langchain-openai en el venv del proyecto Aether-Agents. Estas dependencias se añadirán al requirements.txt del directorio src/.

WARNING

Alcance del MVP: Este plan implementa 3 workflows predefinidos (no dinámicos). Hermes selecciona cuál usar según la tarea, pero no puede "diseñar grafos" sobre la marcha. Eso sería una iteración futura.

IMPORTANT

¿Dónde vive el código? Todo el código nuevo se crea dentro del repositorio Aether-Agents (en src/olympus/), no dentro del SDK ~/.hermes/hermes-agent/. Olympus es parte de tu proyecto, no del upstream.

[NEW] 
state.py
Define el TypedDict que será el estado compartido del grafo. Todos los nodos leen y escriben a este estado.

python
from typing import TypedDict, Annotated, Optional
from langgraph.graph.message import add_messages
class WorkflowState(TypedDict):
    # Prompt original del usuario
    user_prompt: str
    # Contexto acumulado (design docs, research, etc.)
    context: str
    # Código generado (por Hefesto)
    code: str
    # Resultado de auditoría (por Athena)
    audit_result: str
    # ¿La auditoría pasó?
    audit_passed: bool
    # Resultado de investigación (por Etalides)
    research: str
    # Mensajes internos del grafo (log)
    messages: Annotated[list, add_messages]
    # Número de ciclos de revisión completados
    review_cycles: int
    # Máximo de ciclos permitidos antes de forzar salida
    max_review_cycles: int
    # Respuesta final consolidada
    final_response: str
    # PROJECT_ROOT para que cada nodo sepa dónde operar
    project_root: str
[NEW] 
nodes.py
Define las funciones-nodo del grafo. Cada nodo:

Abre sesión con un Daimon vía ACPManager
Envía un prompt contextualizado
Espera la respuesta (via wait)
Escribe el resultado en el estado
python
async def node_design(state: WorkflowState, acp: ACPManager) -> dict:
    """Nodo Hermes: genera diseño/plan basado en el prompt del usuario."""
    ...
async def node_implement(state: WorkflowState, acp: ACPManager) -> dict:
    """Nodo Hefesto: implementa código basado en el diseño y audit feedback."""
    ...
async def node_audit(state: WorkflowState, acp: ACPManager) -> dict:
    """Nodo Athena: audita el código, marca audit_passed=True/False."""
    ...
async def node_research(state: WorkflowState, acp: ACPManager) -> dict:
    """Nodo Etalides: investiga un tema y devuelve hallazgos."""
    ...
def should_retry_implementation(state: WorkflowState) -> str:
    """Edge condicional: si audit_passed=False y cycles < max, volver a implement."""
    if not state["audit_passed"] and state["review_cycles"] < state["max_review_cycles"]:
        return "implement"
    return "finalize"
async def node_finalize(state: WorkflowState, acp: ACPManager) -> dict:
    """Consolida resultados, actualiza .eter/, genera resumen para Hermes."""
    ...
[NEW] 
definitions.py
Contiene los 3 workflows predefinidos como funciones que construyen grafos LangGraph:

Workflow	Nodos	Descripción
dev_and_audit	design → implement ↔ audit → finalize	Desarrollo + auditoría con ciclo de revisión
research_and_implement	research → design → implement → finalize	Investigación profunda → implementación
full_pipeline	research → design → implement ↔ audit → finalize	Pipeline completo con investigación y auditoría
Cada función retorna un CompiledStateGraph listo para ejecutar.

[NEW] 
runner.py
El motor de ejecución. Recibe un nombre de workflow + prompt, construye el estado inicial, ejecuta el grafo, y retorna el resultado.

python
class WorkflowRunner:
    def __init__(self, registry: OlympusRegistry, acp_manager: ACPManager):
        ...
    async def run(
        self,
        workflow_name: str,
        prompt: str,
        project_root: str,
        max_review_cycles: int = 3,
    ) -> dict:
        """Ejecuta un workflow completo y retorna el resultado."""
        ...
Características clave:

Timeout global: 10 minutos por defecto (configurable)
Checkpointing: Usa MemorySaver de LangGraph para permitir inspección del estado
Error handling: Si un nodo falla, marca el estado como error y retorna un resumen parcial

Flujo de Ejecución Detallado
Ejemplo: dev_and_audit
1. Hermes recibe: "Construye un script de login seguro con JWT"
2. Hermes decide usar run_workflow(workflow="dev_and_audit", prompt="...")
3. Olympus invoca WorkflowRunner:
   
   Estado inicial:
   {user_prompt: "...", code: "", audit_passed: false, review_cycles: 0, max: 3}
   
   → Nodo "design" (internamente abre sesión con sí mismo vía ACP):
     Genera DESIGN.md con arquitectura del script
     Estado: {context: "<diseño>", ...}
   
   → Nodo "implement" (abre sesión con Hefesto):
     Prompt: "PROJECT_ROOT: /path\nCONTEXT: <diseño>\nTASK: Implementa..."
     Estado: {code: "<código python>", ...}
   
   → Nodo "audit" (abre sesión con Athena):
     Prompt: "PROJECT_ROOT: /path\nCODE: <código>\nTASK: Audita seguridad..."
     Estado: {audit_result: "SQLi vulnerability en línea 42", audit_passed: false}
   
   → Edge condicional: audit_passed=false, cycles=0 < max=3 → "implement"
   
   → Nodo "implement" (segunda iteración):
     Prompt incluye el feedback de Athena
     Estado: {code: "<código corregido>", review_cycles: 1}
   
   → Nodo "audit" (segunda iteración):
     Estado: {audit_result: "OK - no vulnerabilities", audit_passed: true}
   
   → Edge condicional: audit_passed=true → "finalize"
   
   → Nodo "finalize":
     Consolida resultado, actualiza .eter/
     Estado: {final_response: "Código aprobado tras 2 ciclos de revisión..."}
4. Olympus retorna el final_response a Hermes
5. Hermes presenta el resultado al usuario
Open Questions
IMPORTANT

¿Quieres que el nodo "design" use a Hermes mismo (vía ACP) o que simplemente use el prompt del usuario directamente? Usar ACP para Hermes sería más elegante pero añade complejidad (auto-referencia). La alternativa es que el "diseño" sea una transformación directa del prompt sin LLM.

IMPORTANT

¿El max_review_cycles default de 3 te parece correcto? Un valor alto consume más recursos pero garantiza mejor calidad. Un valor de 2 es más conservador.
