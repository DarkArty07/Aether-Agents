# Multi-Instance CLI Profile Flag — `hermes --profile` Does NOT Work; Use `HERMES_HOME` + Python Module

When a single machine runs more than one hermes-agent instance (e.g. Prometeo + Aether-Agents), the same `hermes` binary in `$PATH` may be a wrapper that defaults to one instance. **As of June 2026, `hermes --profile` is BROKEN** in hermes-agent v0.14.0/v0.15.x — the flag appears in `--help` but is not honored by `hermes_cli.main`. The verified working pattern is to set `HERMES_HOME` as an environment variable and invoke the python module directly.

> **CRITICAL UPDATE (2026-06-05):** Earlier versions of this reference and the matching Pitfall #15 in `aether-agents-orchestration` documented `hermes --profile prometeo mcp list` as the working form. **It is not.** The user verified empirically that this command returns `Error: Profile 'prometeo' does not exist. Create it with: hermes profile create prometeo` even when the profile directory exists and the gateway for that profile is running. The skill content was wrong. This rewrite reflects what actually works.

## The Symptom (Chris, 2026-06-05)

```bash
$ hermes mcp test magnific
  ✗ Server 'magnific' not found in config.
  Available: context7, olympus_v3, graphify

$ hermes mcp -p prometeo test magnific
Error: Profile 'prometeo' does not exist. Create it with: hermes profile create prometeo

$ hermes --profile prometeo mcp test magnific
Error: Profile 'prometeo' does not exist. Create it with: hermes profile create prometeo
```

The user knows Prometeo exists, `magnific` is in `~/.prometeo/profiles/prometeo/config.yaml`, and `hermes-gateway-prometeo.service` is running. The CLI refuses to read that config.

## Root Cause — Verified in Source and CLI Shape

### 1. The wrapper is instance-locked

`~/.local/bin/hermes` (on Chris's machine) is a generated wrapper:
```bash
#!/bin/bash
# Aether Agents wrapper — auto-generated by setup.sh v0.11.1
# Points to .venv-hermes/bin/hermes (default profile)
export HERMES_HOME="/home/prometeo/Aether-Agents/home"
exec "/home/prometeo/Aether-Agents/home/.venv-hermes/bin/hermes" "$@"
```

It hardcodes `HERMES_HOME` to Aether BEFORE launching the binary. The binary itself (`/home/prometeo/Aether-Agents/home/.venv-hermes/bin/hermes`) is also a wrapper that resolves to `python -m hermes_cli.main "$@"`. There is no `--profile` handling anywhere in the chain.

### 2. `hermes --profile` is in `--help` but is NOT a registered argparse argument

`hermes --help` lists `--profile PROFILE` among the global flags, but at runtime `hermes_cli.main` does not register it on its argparse parser. The flag falls through and triggers the "Profile does not exist" error from a profile-creation wizard. This is a hermes-agent bug.

**The same applies to `-p`** — there is no short-form. Both forms silently fail the same way.

### 3. What actually selects the instance: the `HERMES_HOME` env var

`hermes_cli.main` (and `mcp_tool._load_mcp_config`) reads from `$HERMES_HOME`. The systemd services set this in their `[Service] Environment=`:
- `hermes-gateway.service` → `HERMES_HOME=/home/prometeo/Aether-Agents/home`
- `hermes-gateway-prometeo.service` → `HERMES_HOME=/home/prometeo/.prometeo/profiles/prometeo`

The gateway for each profile works because systemd sets the right env var. The CLI does NOT, because the wrapper hardcodes the Aether path.

## Verified Working Commands (June 2026, empirically confirmed)

```bash
# Aether (default wrapper, no env var needed)
hermes mcp list
# → context7, olympus_v3, graphify

# Prometeo (set HERMES_HOME, invoke python module directly — bypass wrapper)
HERMES_HOME=/home/prometeo/.prometeo/profiles/prometeo \
  /home/prometeo/Aether-Agents/home/.venv-hermes/bin/python -m hermes_cli.main mcp list
# → magnific, freepik, reddit, todoist, clio-fca, context7, ...

# Prometeo — test one MCP
HERMES_HOME=/home/prometeo/.prometeo/profiles/prometeo \
  /home/prometeo/Aether-Agents/home/.venv-hermes/bin/python -m hermes_cli.main mcp test magnific

# Prometeo — force OAuth re-auth
HERMES_HOME=/home/prometeo/.prometeo/profiles/prometeo \
  /home/prometeo/Aether-Agents/home/.venv-hermes/bin/python -m hermes_cli.main mcp login magnific
```

**Optional wrapper for convenience** (one-liner alias, no env var prefix each time):

```bash
cat > ~/.local/bin/hermes-prometeo <<'EOF'
#!/bin/bash
export HERMES_HOME=/home/prometeo/.prometeo/profiles/prometeo
exec /home/prometeo/Aether-Agents/home/.venv-hermes/bin/python -m hermes_cli.main "$@"
EOF
chmod +x ~/.local/bin/hermes-prometeo
```

Then: `hermes-prometeo mcp list` from anywhere. Note: this shares the venv with Aether (Prometeo config points its `bin/python` at the Aether venv because the framework is shared), which is by design — Prometeo is a separate profile directory but uses the same installed hermes-agent code.

## Diagnostic Procedure — Always Run This First

When "MCP X not found" or "Profile does not exist" comes back, the cause is almost always: wrong instance, not wrong config.

```bash
# 1. What instance does the wrapper point to?
cat ~/.local/bin/hermes | grep HERMES_HOME

# 2. What instance is THIS shell's binary?
which hermes
ls -la $(which hermes)

# 3. Confirm the MCP exists in the target config
grep -A 5 "mcp_servers:" ~/.prometeo/profiles/prometeo/config.yaml | grep magnific
grep -A 5 "mcp_servers:" ~/Aether-Agents/home/config.yaml | grep magnific

# 4. Check the gateway is running for the right instance
systemctl --user status hermes-gateway.service
systemctl --user status hermes-gateway-prometeo.service
```

If step 1 shows the wrapper points to Aether but you want Prometeo → use the `HERMES_HOME=... python -m hermes_cli.main ...` form.

## Why This Is a Hard Rule, Not a Tip

- The error message "Profile 'prometeo' does not exist" is misleading: the profile DOES exist as a directory (`~/.prometeo/profiles/prometeo/`) and as a config file. The bug is that the CLI flag isn't wired up.
- Trying to fix this by running `hermes profile create prometeo` would create a NEW profile (likely under `~/Aether-Agents/home/profiles/prometeo/`) and still not solve the actual problem. **Do not run that command.**
- The "not found" response lists only Aether MCPs because the wrapper is already locked to Aether before the subcommand runs.

## No-Wrapper Alternative (Preferred When Possible)

When the user controls the entire machine and can dedicate a venv per instance, the cleanest pattern avoids wrappers entirely by giving each instance its own venv with its own default profile:

```
Instance A: ~/.venv-a/bin/hermes, HERMES_HOME=~/.data-a, config at ~/.data-a/config.yaml
Instance B: ~/.venv-b/bin/hermes, HERMES_HOME=~/.data-b, config at ~/.data-b/config.yaml
```

Each gets a symlink in `~/.local/bin/` (e.g., `hermes-a` → `~/.venv-a/bin/hermes`). No wrapper scripts, no shared venv, no `-p` flag needed. The HERMES_HOME is set in `.bashrc` for the primary instance; secondary instances use their dedicated symlink name. See SKILL.md §"Standard Setup Without Wrappers" for the full pattern and decision tree.
