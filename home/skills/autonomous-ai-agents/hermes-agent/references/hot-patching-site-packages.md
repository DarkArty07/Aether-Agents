# Hot-Patching Hermes Agent site-packages

When Hermes Agent behavior needs modification at the runtime level (not config level), the installed `site-packages/` files can be patched directly. This pattern is used for OAuth noise suppression and similar fixes where upstream changes would take too long.

## When to Use This Pattern

- A Hermes Agent module has a behavior that is undesirable (noisy output, blocking call, wrong default)
- The config layer (`config.yaml`, `.env`, `SOUL.md`) cannot control the behavior
- The fix is small and localized (one function, one conditional)
- You cannot wait for an upstream release

**Do NOT hot-patch if** the fix can be achieved via `config.yaml`, `.env`, or environment variables. Hot-patching is invasive and lost on `pip install --upgrade`.

## The Pattern (4 Steps)

### 1. Find the file

```bash
find /path/to/venv -name "mcp_oauth.py" -path "*/site-packages/*"
# → /home/prometeo/.venv-hermes/lib/python3.11/site-packages/tools/mcp_oauth.py
```

### 2. Read the function around the target line (use `read_file` with offset)

```python
async def _redirect_handler(authorization_url: str) -> None:
    msg = (
        f"\n  MCP OAuth: authorization required.\n"
        f"  Open this URL in your browser:\n\n"
        f"    {authorization_url}\n"
    )
    print(msg, file=sys.stderr)       # ← Line to suppress
```

### 3. Apply the patch — insert an early return before the print

```python
async def _redirect_handler(authorization_url: str) -> None:
    msg = (
        f"\n  MCP OAuth: authorization required.\n"
        f"  Open this URL in your browser:\n\n"
        f"    {authorization_url}\n"
    )

    # Suppress OAuth prompt when we have cached tokens — the CLI shouldn't
    # show authorization URLs on every startup when tokens already exist.
    _tokens_dir = os.environ.get("HERMES_HOME", os.path.expanduser("~/.hermes"))
    _tokens_path = os.path.join(_tokens_dir, "mcp-tokens")
    if os.path.isdir(_tokens_path) and any(f.endswith(".json") for f in os.listdir(_tokens_path)):
        return  # Silently skip — the callback handler will time out quickly

    print(msg, file=sys.stderr)
```

Use `patch` tool with sufficient context around the `old_string` to ensure uniqueness.

### 4. Delete the `__pycache__` — otherwise Python uses the cached bytecode

```bash
rm -rf /path/to/venv/lib/python*/site-packages/tools/__pycache__/mcp_oauth*
```

**Why this is required**: Python's import system caches compiled bytecode in `__pycache__/`. Editing the `.py` file alone does NOT invalidate the cache if the file's `mtime` doesn't change deterministically. Explicitly deleting the cache files forces recompilation on the next import.

### 5. Restart the affected service

```bash
systemctl restart hermes-gateway-<instance>
```

## Applied Examples

### Example A — Suppress OAuth URL print on CLI startup

**File:** `tools/mcp_oauth.py` → function `_redirect_handler()`

**Problem:** Every CLI startup triggers Magnific OAuth redirect even though valid cached tokens exist. The print shows "MCP OAuth: authorization required" + full authorization URL to stderr.

**Fix:** Early-return check for cached `.json` files in `$HERMES_HOME/mcp-tokens/` (or `~/.hermes/mcp-tokens/` as fallback) before the `print(msg, ...)` line.

**Coupled with `_wait_for_callback()` timeout fix:** In the same file, `_wait_for_callback()` was also patched to use a 5-second timeout when cached tokens exist (instead of the default 300s). Together these two patches make the CLI startup silent when tokens are present.

### Example B — Changing behavior of any other `tools/` module

Same pattern: modify `.py`, delete `__pycache__/` for that module, restart.

## Caveats

- **Lost on upgrade:** `pip install --upgrade hermes-agent` overwrites the patched file.
- **venv-local:** The patch only affects one venv. If you have multiple Hermes instances (Prometeo, Aether-Agents, etc.), each must be patched independently.
- **Token directory name matters:** The SDK uses `mcp-tools/` per `_get_token_dir()`, but the patch checks `mcp-tokens/` (a common alias in documentation and `.env` comments). If the actual token directory is `mcp-tools/`, adjust the patch path accordingly. Verify with:

```bash
grep -n '_get_token_dir\|mcp-tools' \
  $HERMES_HOME/venv/lib/python*/site-packages/hermes_agent/tools/mcp_oauth.py 2>/dev/null | head -3
```

- **Cross-WSL patching:** When the target venv is on a different WSL distro, use `wsl.exe -d <DISTRO> -u <USER> -- bash -c` to read/write files. Write a Python patch script to a shared location (e.g. `/mnt/c/Users/.../Desktop/`) and execute it on the target distro to avoid bash quoting issues. See `references/cross-wsl-diagnostics.md` for stdin heredoc technique.
