### 1c. olympus_v3 FK Constraint Bug Fix (diagnosed and patched same session)

After downgrade to v0.15.2, `delegate()` still returned `last_turn: null` even though Daimons produced `stop_reason=end_turn`. Root cause was **two separate issues**:

**Issue A (FIXED): FK constraint failure.** The `olympus_v3_hooks` plugin had NO `on_session_start` hook. When `post_llm_call` tried to INSERT INTO `turns(session_id, ...)`, the `sessions` table was empty (no FK parent row), causing `FOREIGN KEY constraint failed`. The error was caught and silently logged as a WARNING — all turn data was lost.

The `aether_hooks` plugin has `on_session_start` but writes to `aether.db`, not `olympus_v3.db`. The `acp_manager` does call `insert_session()` but uses async aiosqlite, and the WAL write isn't guaranteed visible to the sync sqlite3 connection before the hook fires.

**Fix (3 files):**
1. `src/olympus_v3/olympus_v3_hooks/hooks.py` — Added `on_session_start()` hook that creates session row in `olympus_v3.db` via `OlympusDBSync.insert_session()`. Registered in `register()`.
2. `src/olympus_v3/db.py` — Added `OlympusDBSync.insert_session()` method (sync alternative to async `OlympusDB.insert_session()`). Added `PRAGMA wal_checkpoint = TRUNCATE` in `ensure_tables()` to force WAL checkpoint for async→sync visibility.
3. Updated module docstring to list 5 hooks (was 4).

**Verification:**
```bash

  "SELECT session_id, agent, status FROM sessions; SELECT turn_num, role, substr(content,1,80) FROM turns;"
# Expected: 1 session (completed), 1+ turns (assistant, with content)

# Check agent.log for on_session_start

# Expected: "olympus_v3.hooks: on_session_start: session <uuid> created for agent"
```

**Issue B (DIAGNOSED): Zombie olympus_v3.server processes.** After gateway restart, multiple `olympus_v3.server` processes exist. Old processes hold file descriptors to the DELETED database file (`olympus_v3.db (deleted)` in `/proc/PID/fd`). MCP requests hitting an old process return stale/empty data — `last_turn: null`, `thoughts: 0`, `messages: 0`. The DB itself is correct (verified via direct `OlympusDB.get_session_progress()` call).

**Diagnostic:** `ls -la /proc/$(pgrep -f olympus_v3.server)/fd | grep deleted` — any `(deleted)` entries indicate zombie processes holding stale file handles.

**Fix (must run before EVERY gateway restart):**
```bash
pkill -f 'olympus_v3.server'
systemctl --user restart hermes-gateway.service
# Verify: exactly 1-2 processes, no (deleted) FDs
ps aux | grep olympus_v3.server | grep -v grep
```

This is NOT a protocol mismatch in the code — it's a process lifecycle management bug. The gateway spawns fresh MCP server processes on restart but does not kill the old ones.

## Known Issues After Refactoring

### v0.15.2 ACP Response Propagation (Zombie Process Bug — DIAGNOSED)
After downgrading from v0.16.0 to v0.15.2, the `LimitOverrunError` crash is fixed and the olympus_v3 FK bug is fixed. Daimons initialize correctly, produce `stop_reason=end_turn`, and the DB is populated with sessions and turns. However, `talk_to(action="delegate")` returns `last_turn: null, tool_calls: 0` via the MCP layer when zombie olympus processes exist.

**Root cause (identified 2026-06-09):** `systemctl --user restart hermes-gateway.service` spawns new `olympus_v3.server` processes but does NOT kill old ones. Old processes hold file descriptors to deleted DB files (`/proc/PID/fd` shows `(deleted)`). MCP requests routed to old processes return stale/empty data.

**Fix:** Kill ALL olympus processes before restarting the gateway:
```bash
pkill -f 'olympus_v3.server'
systemctl --user restart hermes-gateway.service
```
See Pitfall #24 in SKILL.md for full diagnostic procedure.

## Gateway Restart Required After Version Change

After ANY `pip install` that changes `hermes-agent` version (upgrade or downgrade), the gateway MUST be restarted:
```bash
hermes gateway restart
# Or: systemctl --user restart hermes-gateway.service
```
Without restart, the gateway process continues running the OLD version loaded in memory. The old processes (olympus_v3, Daimons) also keep the old code. Only a full restart ensures all components use the new version.
